## 1. 공통 규약

### Base Path

```
/api/v1
```

### 응답 포맷

기존 구현된 `ApiResponse<T>`를 그대로 사용.

```json
{ "success": true, "data": { ... }, "error": null }
{ "success": false, "data": null, "error": { "code": "...", "message": "..." } }
```

### 인증

```
Authorization: Bearer {accessToken}
```

배포 완료된 JWT 인증 체계 그대로. (이전 명세서의 🟡 가정 표시는 이제
✅ 확정으로 갱신)

### 캡처 유형 (CardType)

✅ **확정** — 기존 코드베이스 `CardType` enum을 그대로 사용한다.
(JOB, SHOPPING, PLACE, SCHEDULE, KNOWLEDGE, CONTENT, BENEFIT, RECORD, ETC)

화면 표기(정보·지식, 책·콘텐츠)와 enum 값(KNOWLEDGE, CONTENT)의 한글-영문
매핑은 FE와 별도 상수표로 공유하면 되고, 서버 enum 자체는 변경하지 않는다.

### 페이지네이션

검색 결과(06-02)는 20개 단위 무한 스크롤. 그 외 목록(홈 최근 3개,
즐겨찾기 3개, 자주 저장 유형 4개, 보관함 목록)은 개수 제한 없이 전체
반환(세로 스크롤).

```json
{ "items": [...], "hasNext": true, "nextPage": 1 }
```

### 시간 표기 공통 규칙

- 정리 완료 시점 = `InfoCard.createdAt` (캡처는 정리 완료와 동시에
  생성되는 아키텍처이므로 별도 `organizedAt` 컬럼 없이 `createdAt`
  재사용 — 🟡 확인 필요, 아래 7번 데이터 모델 참고)
- 클라이언트가 상대 시간("방금 전", "N분 전" 등) 및 절대 날짜
  포맷("M월 D일 정리", "YYYY.MM.DD")을 직접 계산. 서버는 ISO-8601
  UTC(`Instant`)만 내려준다.

## 2. 업로드 · 정리 플로우

### 🔴 확인 필요 — 이미지 업로드 방식

ADR-0010에서 미결정으로 남겨둔 부분입니다. 이 명세는 **presigned URL
방식**(클라이언트가 S3에 직접 업로드, 서버는 URL만 발급)으로 설계했습니다
— t3.small EC2가 이미지 바이너리를 직접 처리하지 않아도 되기 때문입니다.
다른 방식(서버 경유 업로드)을 원하시면 이 섹션만 다시 설계하면 됩니다.

### 2-1. 업로드용 presigned URL 발급

```
POST /captures/upload-urls
```

**요청**

```json
{ "count": 5 }
```

- `count`: 1~20 (최대 장수 20 임시 확정, 이번 스프린트 고정값)

**응답**

```json
{
  "uploads": [
    { "imageKey": "captures/{userId}/{uuid}.jpg", "uploadUrl": "https://..." }
  ]
}
```

**에러**

| 코드 | 상황 |
| --- | --- |
| INVALID_INPUT | count가 1~20 범위를 벗어남 |

### 2-2. 정리 시작 (배치 생성)

```
POST /captures/organize
```

클라이언트가 2-1에서 받은 `imageKey`들로 S3 업로드를 마친 뒤 호출.

**요청**

```json
{ "imageKeys": ["captures/{userId}/{uuid}.jpg", "..."] }
```

**응답**

```json
{ "batchId": 123, "totalCount": 5, "status": "PROCESSING" }
```

서버는 각 이미지를 **개별 처리 후 개별 커밋**한다(설계서 03-C 정책과
일치). 하나가 실패해도 나머지는 계속 처리된다.

**에러**

| 코드 | 상황 |
| --- | --- |
| INVALID_INPUT | imageKeys가 비어있거나 20장 초과 |
| ORGANIZE_IN_PROGRESS | 이미 진행 중인 배치가 있는 상태에서 재요청 (설계서 "진행 중 새 정리 시작 불가" 정책) |

### 2-3. 정리 상태 조회 (폴링)

```
GET /captures/organize/{batchId}/status
```

**응답**

```json
{
  "batchId": 123,
  "status": "PROCESSING",
  "totalCount": 5,
  "successCount": 3,
  "failCount": 0
}
```

`status`: `PROCESSING` | `COMPLETED`(전체 성공) | `PARTIAL_FAILED`(일부
실패) | `FAILED`(전체 실패) | `CANCELLED`

### 2-4. 정리 취소

```
POST /captures/organize/{batchId}/cancel
```

**응답**: `204 No Content`

취소 시점까지 이미 완료(커밋)된 캡처는 그대로 저장 유지, 아직 처리
안 된 나머지만 중단한다(설계서 03-C 정책).

🟡 **확인 필요**: "취소 시점에 이미 완료된 항목 처리"가 설계서에도
"개발 협의 필요"로 명시되어 있음 — 위 방식(완료분 유지)으로 진행해도
되는지 확인 필요.

### 2-5. 이탈 후 복귀 시 결과 확인 (홈 진입용)

```
GET /captures/organize/pending-result
```

홈 화면 진입/체류 시점에 호출해, 사용자가 확인하지 못한 완료된 배치가
있는지 조회. 있으면 토스트 노출 후 아래 ack API 호출.

**응답**

```json
{ "batchId": 123, "status": "PARTIAL_FAILED", "successCount": 4, "failCount": 1 }
```

없으면 `data: null`.

```
POST /captures/organize/{batchId}/ack
```

토스트 노출 확인 처리(재노출 방지). **응답**: `204 No Content`

🟡 **확인 필요**: 앱 미실행 상태의 푸시 알림은 FCM 연동이 아직 RECAP에
구축되어 있지 않습니다. 이번 스프린트는 **토스트(홈 진입 시 조회)까지만
포함**하고 푸시는 별도 스프린트로 미루는 것을 제안합니다.

### 실패 케이스 정의

🟡 설계서에 "대표 실패 케이스: AI 분석 실패, 서버/클라이언트 통신
실패 [상세 리스트업 개발 협의 필요]"로 명시되어 있음. 최소한 아래
두 코드로 시작 제안:

| 코드 | 상황 |
| --- | --- |
| IMAGE_ANALYSIS_FAILED | AI 분석 자체 실패 |
| IMAGE_UPLOAD_VERIFICATION_FAILED | S3 업로드 확인 실패(클라이언트가 imageKey를 보냈으나 실제 파일이 없음 등) |

## 3. 홈 (02-01)

```
GET /home/summary
```

**응답**

```json
{
  "recentCaptures": [
    { "captureId": 1, "title": "...", "summary": "...", "typeCode": "SCHEDULE", "thumbnailUrl": "...", "isFavorite": false, "organizedAt": "2026-07-13T02:00:00Z" }
  ],
  "favorites": [ /* 위와 동일 구조, 최대 3개 */ ],
  "topTypes": [
    { "typeCode": "SHOPPING", "count": 12, "representativeThumbnailUrl": "..." }
  ],
  "hasAnyCapture": true
}
```

- `recentCaptures`: 최근 30일 이내 정리 완료, 최신순 최대 3개. 30일
  이내 캡처가 없으면 빈 배열(클라이언트가 유도 문구 노출).
- `favorites`: 즐겨찾기 등록 시점 최신순 최대 3개.
- `topTypes`: 유형별 캡처 수 많은 순 최대 4개. 동수면 최근 정리
  캡처가 포함된 유형 우선.
- `hasAnyCapture`: 전체 캡처가 0개인지(빈 상태 S-02-01 노출 판단용).

정리 결과 안내(토스트)는 2-5의 별도 API로 분리했습니다 — 홈 요약과
관심사가 달라서 하나로 합치지 않는 게 낫다고 판단했습니다.

## 4. 보관함 (05-01, 05-02)

### 4-1. 즐겨찾기 탭

```
GET /storage/favorites
```

**응답**: `{ "count": 3, "items": [ ... ] }` — 즐겨찾기 등록 시점
최신순 전체, 정렬 옵션 없음(고정).

### 4-2. 기타 탭

```
GET /storage/etc?sort=latest
```

`sort`: `latest`(기본) | `oldest`. **응답**: 위와 동일 구조.

### 4-3. 유형별 보기 — 유형 목록

```
GET /storage/types
```

**응답**

```json
{
  "folderCount": 8,
  "types": [
    { "typeCode": "SHOPPING", "count": 12, "representativeTitles": ["택배 반품 절차", "노트북 가격 비교"] }
  ]
}
```

캡처가 1개 이상인 유형만 반환(최대 8개, 기타 제외). 개수 많은 순,
동수면 최근 정리 캡처 포함 유형 우선. `representativeTitles`는
리스트형 뷰용 — 최근 정리 순 최대 2개(24자 말줄임은 클라이언트 처리).

### 4-4. 유형별 보기 — 특정 유형 상세 (05-02)

```
GET /storage/types/{typeCode}/captures?sort=latest
```

`sort`: `latest`(기본) | `oldest`. **응답**: 4-1과 동일 구조 +
`typeCode`, `count`.

## 5. 정보카드 상세 (07-01)

### 5-1. 상세 조회

```
GET /captures/{captureId}
```

**응답**

```json
{
  "captureId": 1,
  "typeCode": "SCHEDULE",
  "title": "제주 숙소 예약 정보",
  "summary": "체크인 오후 3시, 8월 1박 예약이 확정된 숙소 정보 요약",
  "body": "Lorem ipsum ...",
  "originalImageUrl": "https://...(presigned GET URL)",
  "isFavorite": false,
  "organizedAt": "2026-07-13T02:00:00Z"
}
```

`originalImageUrl`은 presigned URL이라 만료될 수 있음 — 클라이언트가
로딩 실패(S-07-01) 시 이 API를 재호출하면 새 URL을 받는다.

### 5-2. 즐겨찾기 토글

```
PATCH /captures/{captureId}/favorite
```

**요청**

```json
{ "isFavorite": true }
```

🔴 **기존 코드 수정 필요**: 현재 `InfoCard.toggleFavorite()`는
호출할 때마다 상태를 뒤집는 방식(비멱등)이라, 네트워크 재시도 시
의도치 않게 상태가 반대로 반영될 위험이 있습니다. **`markFavorite(boolean)`처럼
원하는 상태를 명시적으로 받는 멱등적 메서드로 교체가 필요합니다.**
이 API 설계(true/false 명시)는 그 전제로 작성했습니다.

**응답**: `204 No Content`

### 5-3. 삭제 (스코프 확인 필요, 0번 참고)

```

```

**응답**: `204 No Content`. 소프트 삭제 권장(원본 이미지는 S3에서도

## 6. 검색 (06-01, 06-02)

### 6-1. 최근 검색어

설계서에 "최근 검색어는 기기 로컬 저장, 서버 저장 실익 낮음"으로
명시되어 있습니다. **서버 API가 필요 없다고 판단해 이번 명세에서
제외했습니다.** (검색 가능 안내 영역의 "최근 정리 완료 순 예시 캡처
최대 6개"는 `GET /home/summary`의 `recentCaptures`를 재사용하거나
별도 API가 필요할 수 있음 — 🟡 확인 필요.)

### 6-2. 검색 실행

```
GET /search?q={검색어}&scope=all&page=0&size=20
```

| 파라미터 | 설명 |
| --- | --- |
| `q` | 1~100자, 서버에서 trim + 중간 연속 공백 정규화 |
| `scope` | `all`(전체) \| `favorite` \| `etc` \| `type` |
| `typeCode` | `scope=type`일 때만 필수 |

**응답**

```json
{
  "count": 3,
  "hasNext": false,
  "items": [
    {
      "captureId": 1,
      "typeCode": "SCHEDULE",
      "titleHighlighted": "제주 <mark>숙소</mark> 예약 정보",
      "summaryHighlighted": "체크인 오후 3시, 8월 1박 예약 확정",
      "isFavorite": true,
      "organizedAt": "2026-07-13T02:00:00Z"
    }
  ]
}
```

- 검색 대상: `title`, `summary`, `body`, `extractedText`(OCR, 응답엔
  노출 안 함). `typeCode`는 검색 대상 아님.
- 정렬: 제목 매칭 > 요약 매칭 > 본문/추출텍스트 매칭 순, 동순위 내
  정리 완료 최신순.
- 하이라이트: 제목·요약만 `<mark>` 마크업, 본문/추출텍스트는 매칭
  여부만 결과 포함에 반영하고 마크업 없음.
- 삭제된 캡처는 검색 대상 제외.

🟡 **구현 난이도 참고**: 이 정렬·하이라이트 로직은 단순 `LIKE`
검색보다 복잡합니다(매칭 위치별 우선순위 정렬 + 부분 문자열
하이라이트 삽입은 애플리케이션 레이어에서 직접 구현 필요). 일정에
이 난이도가 반영되어 있는지 재확인 권장.

**에러**

| 코드 | 상황 |
| --- | --- |
| INVALID_INPUT | q가 비었거나 100자 초과, scope=type인데 typeCode 누락 |

## 7. 데이터 모델 (갱신)

### InfoCard (변경분만)

```
title              varchar(30)   not null
summary            varchar(80)   nullable
body               text          nullable
originalImageKey   varchar(255)  not null   -- S3 오브젝트 키 (ADR-0010)
extractedText       text          nullable   -- OCR 원문, 검색 전용, 응답 미노출
favoritedAt        datetime      nullable   -- 즐겨찾기 등록 시점 (정렬용, 신규)
batch              FK(OrganizeBatch) nullable
```

🟡 **확인 필요**: `title`/`summary`/`body`를 CardSummary(다형성 JSON)
안에 넣을지, 이렇게 평범한 컬럼으로 뺄지. 이 문서 서론에서 말씀드린
대로 **평범한 컬럼 방식을 권장**합니다(검색 정렬/하이라이트가 SQL
레벨에서 훨씬 단순해짐). `CardSummary` 인터페이스와 기존 구현체는
코드에 남기되 이번 스프린트엔 미사용.

기존 `keywords`(List<String>) 필드는 이번 명세의 검색 정책과 역할이
겹칩니다(제목/요약/본문/추출텍스트로 검색). **`keywords` 필드는
이번 스프린트에서 안 쓰는 것으로 판단** — 유지할 이유가 있다면 확인
필요.

### OrganizeBatch (신규)

```
id            PK
user          FK(User)
status        PROCESSING | COMPLETED | PARTIAL_FAILED | FAILED | CANCELLED
totalCount    int
successCount  int
failCount     int
acknowledged  boolean   -- 홈 토스트 확인 여부 (2-5 ack API)
createdAt     Instant
```
