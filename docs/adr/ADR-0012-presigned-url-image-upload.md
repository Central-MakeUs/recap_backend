# ADR-0012: 원본 이미지 업로드는 Presigned URL 방식으로 확정

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-13 |
| 관련 | [ADR-0010](ADR-0010-original-image-s3-storage.md)(후속 미결정 항목 해결) |

## 맥락 (Context)

ADR-0010에서 원본 이미지를 S3에 저장하기로 결정했으나, "클라이언트가
S3에 직접 업로드할지, 서버를 경유할지"는 미결정으로 남겨뒀다.

RECAP은 EC2 t3.small 단일 인스턴스로 운영 중이다. 이미지 바이너리를
서버가 직접 받아 S3로 다시 전달하는 구조(서버 경유)는 t3.small의
제한된 네트워크 대역폭·CPU를 소비하고, 최대 20장(임시 확정)을 한
번에 업로드하는 이번 기능의 특성상 부담이 커질 수 있다.

## 결정 (Decision)

클라이언트가 S3에 **직접 업로드**한다.

```
1. POST /captures/upload-urls { count: N }
   → 서버가 S3 presigned PUT URL을 N개 발급
2. 클라이언트가 각 URL로 S3에 직접 이미지 업로드 (서버 경유 없음)
3. POST /captures/organize { imageKeys: [...] }
   → 업로드 완료된 imageKey들로 정리 시작 요청
```

서버는 이미지 바이너리를 한 번도 직접 다루지 않는다 — presigned URL
발급과, 이후 정리 단계에서 S3에 있는 이미지를 읽어 AI 분석에 넘기는
역할만 한다.

## 고려한 대안 (Considered Options)

1. **서버 경유 업로드 (기각)** — 구현은 단순하지만 이미지 바이너리가
   EC2를 그대로 통과해 t3.small의 대역폭·메모리 부담이 커짐. 최대
   20장 동시 업로드 시 부담이 더 커짐.
2. **Presigned URL 직접 업로드 (채택)** — 서버 부하 없이 확장 가능.
   이미 Android Apple 로그인 검토 시 같은 패턴(RFC 8252 기반 exchange
   code 방식과 유사한 "서버는 URL만 발급, 실제 전송은 클라이언트-외부
   서비스 간" 구조)을 적용한 전례가 있어 팀에 익숙한 패턴.

## 결과 (Consequences)

### 긍정
- 서버가 이미지 바이너리 트래픽을 전혀 처리하지 않아 t3.small에서도
  부담 없음.
- S3로의 업로드 자체는 AWS 인프라가 처리하므로 안정성·확장성이 높음.

### 부정 / 트레이드오프
- presigned URL은 업로드 완료 여부를 서버가 즉시 알 수 없다 —
  클라이언트가 업로드 후 반드시 `/captures/organize`를 호출해야
  하는데, 그 전에 이탈하면 S3에 고아 파일(orphan file)이 남을 수
  있다.
- presigned URL 만료 시간, S3 버킷 CORS 설정 등 추가 구성 필요.

## 후속 / 미결정

- [ ] S3 고아 파일(업로드했지만 organize 호출 안 된 이미지) 정리 정책
      — 예: 24시간 이상 미사용 `captures/{userId}/` 아래 임시 업로드
      건을 배치로 삭제
- [ ] presigned URL 만료 시간 값 확정 (제안: 10분 — 20장 업로드에
      충분한 여유)
- [ ] S3 버킷 CORS 설정 (클라이언트 도메인/앱 스킴 허용) — 네이티브
      앱이라 브라우저 CORS 제약은 없지만, 웹 기반 업로드 테스트 도구
      사용 시 필요할 수 있음
