# ADR-0014: 검색 기능은 MySQL LIKE 기반으로 시작 (전용 검색엔진 도입 보류)

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-19 |
| 관련 | [LLD-0007](../lld/LLD-0007-search-api.md), 이슈 #17 |

## 맥락 (Context)

06-01/06-02(검색) 화면은 캡처의 제목·요약·본문·OCR 추출 텍스트를
대상으로, 매칭 위치별 우선순위 정렬(제목 > 요약 > 본문/추출텍스트)과
동순위 내 최신순 정렬을 요구한다.

검색을 구현하는 기술적 선택지는 크게 세 갈래였다: (1) RDB의 단순
`LIKE` 매칭, (2) MySQL의 FULLTEXT 인덱스, (3) Elasticsearch/OpenSearch
같은 전용 검색엔진 도입. 이 선택은 이번 이슈 하나의 엔드포인트
설계를 넘어, RECAP 검색 기능 전체가 앞으로 어떤 기술 기반 위에 서는지를
정하는 결정이라 LLD가 아니라 ADR로 남긴다.

## 결정 (Decision)

**MySQL `LIKE` 기반 매칭으로 시작한다.** `InfoCardRepository.search()`가
`title`/`summary`/`body`/`extractedText`에 대해 `lower(...) like
lower(concat('%', :q, '%'))`로 매칭하고, 매칭 위치는 JPQL의 `CASE`
표현식으로 우선순위를 매겨 정렬한다(상세는 LLD-0007).

전용 검색엔진이나 FULLTEXT 인덱스는 지금 도입하지 않는다.

## 고려한 대안 (Considered Options)

1. **MySQL FULLTEXT 인덱스 (보류)** — `LIKE '%q%'`보다 성능이 낫고
   한국어 형태소 분석기(N-gram 파서 등)를 붙이면 검색 품질도 개선
   가능하다. 다만 인덱스 설계·형태소 파서 설정이라는 별도 작업이
   필요하고, 지금 유저당 데이터 규모(개인용 스크린샷 정리 앱, 대량
   문서 검색이 아님)에서는 `LIKE`로도 체감 성능 문제가 없을 것으로
   판단해 지금 도입할 실익이 낮다.
2. **Elasticsearch/OpenSearch 등 전용 검색엔진 (기각)** — 매칭 우선순위
   정렬, 하이라이트, 형태소 분석까지 강력하게 지원하지만, RDB 외에
   별도 인프라(운영·동기화 파이프라인)를 t3.small 단일 인스턴스
   환경에 추가로 얹는 셈이라 지금 규모에 명백히 과함(YAGNI). 데이터
   동기화 지연, 추가 운영 비용 같은 부담이 지금 얻는 이득보다 크다.
3. **MySQL LIKE 기반 (채택)** — 추가 인프라 없이 기존 RDB만으로
   매칭·우선순위 정렬·페이지네이션을 전부 구현 가능. `docs/lld/LLD-0007`에
   이미 구현되어 검증까지 마쳤다(COUNT 쿼리 정상 동작 확인 등).

## 결과 (Consequences)

### 긍정
- 추가 인프라·운영 비용 없이 지금 바로 요구사항을 충족.
- 기존 RDB(RDS)만으로 완결되어 배포 파이프라인이 단순하게 유지됨.

### 부정 / 트레이드오프
- `LIKE '%q%'`는 인덱스를 활용하지 못해(앞에 `%`가 붙는 패턴) 데이터가
  많아지면 풀스캔에 가까워질 수 있음.
- 한국어 형태소 분석이 없어, 예를 들어 "카페"로 검색 시 "카페라떼"는
  걸리지만 활용형이 다른 표현(조사가 붙은 형태 등)은 놓칠 수 있음 —
  지금은 단순 부분 문자열 매칭 수준.

## 후속 / 미결정

- [ ] 유저당 데이터 규모가 커지거나(예: 수천~수만 건), 검색 응답
      지연이 실제로 관측되면 FULLTEXT 인덱스 도입을 우선 검토
- [ ] 한국어 검색 품질(형태소 분석 부재로 인한 누락)에 대한 사용자
      피드백이 쌓이면, FULLTEXT N-gram 파서 또는 전용 검색엔진 재검토
- [ ] 위 전환이 필요해지는 시점의 구체적 임계치(데이터 건수, 응답
      시간 등)는 아직 정해두지 않음 — 운영 지표를 관찰하며 판단
