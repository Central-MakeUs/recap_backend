# ADR-0011: InfoCard 표시 필드(title/summary/body)를 평문 컬럼으로 확정

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-13 |
| 관련 | [ADR-0002](ADR-0002-card-summary-json-polymorphism.md)(범위 조정, Superseded 아님) |

## 맥락 (Context)

ADR-0002는 유형마다 요약 필드 구조가 다르다는 전제(채용 = 회사명·
직무명·마감일, 장소 = 장소명·위치·영업시간 등)로 `CardSummary` 다형성
JSON 구조를 설계했다.

이번 스프린트의 화면 설계서(02, 03, 05, 06, 07) 전체를 검토한 결과,
**실제 화면 어디에도 유형별 구조화 필드가 렌더링되지 않는다.** 07-01
(정보카드 상세), 07-02(수정)를 포함해 모든 유형이 동일하게 "유형 뱃지 +
제목 + 한 줄 요약 + 본문"만 노출한다. 즉 ADR-0002가 전제했던 "유형마다
다른 필드 구조"가 이번 스프린트의 실제 요구사항과 맞지 않는다.

또한 검색(06-02)은 제목·요약을 기준으로 매칭 우선순위 정렬과
하이라이트를 수행해야 하는데, 이 값들이 JSON 컬럼 안에 있으면 SQL
레벨에서 정렬·인덱싱이 번거로워진다.

## 결정 (Decision)

`InfoCard`에 `title`(varchar 30, not null), `summary`(varchar 80,
nullable), `body`(text, nullable)를 **평문 컬럼으로 직접 추가**한다.

`CardSummary` 인터페이스와 `CardSummaryConverter`, 기존에 설계했던
유형별 구현체(`JobSummary` 등)는 **코드에서 제거하지 않고 그대로
남긴다.** 다만 이번 스프린트에는 어디에서도 이 다형성 구조를 사용하지
않는다 — 유형별 구조화 필드가 실제로 필요해지는 시점(예: 검색/필터
고도화, 유형별 특화 UI 도입)에 재도입할 수 있는 여지로 남겨둔다.

**본 ADR은 ADR-0002를 Superseded 처리하지 않는다.** ADR-0002의 결정
(다형성 JSON 설계 자체)이 틀렸던 것이 아니라, 이번 스프린트 범위에서
그 결정을 적용할 대상이 없어진 것뿐이다.

## 고려한 대안 (Considered Options)

1. **ADR-0002대로 CardSummary 다형성 JSON 유지 (기각)** — 실제 화면
   요구사항과 맞지 않고, 검색 정렬/하이라이트 구현이 불필요하게
   복잡해짐(JSON 내부 값 기준 정렬은 MySQL에서 추가 작업 필요).
2. **title만 평문 컬럼으로 빼고 summary/body는 JSON 유지 (기각)** —
   검색이 summary/body도 매칭 대상으로 삼아야 해서 절반만 빼는 건
   일관성이 없고, 어차피 이번 스프린트엔 유형별 구조 차이가 전혀
   없으므로 셋 다 평문으로 통일하는 게 단순함.
3. **평문 컬럼 + CardSummary 코드 보존 (채택)** — 지금 당장 필요한
   요구사항에 맞게 단순화하면서, 향후 재도입 가능성을 코드 삭제 없이
   열어둔다.

## 결과 (Consequences)

### 긍정
- 검색 정렬·하이라이트가 일반 컬럼 기준 SQL로 단순하게 구현 가능.
- 유형과 무관하게 모든 캡처가 동일한 필드 구조라 API/DTO가 단순해짐.
- `CardSummary` 관련 코드 자산을 잃지 않음.

### 부정 / 트레이드오프
- `CardSummary`/`CardSummaryConverter`가 당분간 죽은 코드(dead code)로
  남는다 — 나중에 재도입 시점에 실제로 잘 맞물리는지 재검증 필요.
- 유형별 특화 필드가 필요해지면 이번에 추가한 평문 컬럼과 다형성 JSON을
  어떻게 공존시킬지 다시 설계해야 한다.
