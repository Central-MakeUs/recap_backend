# ADR-0002: 카드 요약 필드의 JSON 컬럼 + 다형성 컨버터 저장

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-07 (소급 기록 — 실제 결정은 2026-07 초) |
| 관련 | InfoCard, CardSummary, CardSummaryConverter, ADR-0005 |

## 맥락 (Context)

MVP 재정립 회의에서 목적형 4컬렉션이 폐기되고 "정보카드 + 유형 9개(+기타) +
필터" 구조로 전환되었다. 핵심 제약은 다음과 같다.

- **유형마다 요약 필드 구조가 다르다** (채용: 회사명·직무·마감일 / 쇼핑:
  제품명·가격·브랜드 / 장소: 장소명·위치·영업시간 …).
- 회의록에 "유형 명칭·개수는 팀 논의로 조정 가능"이 명시됨 — 유형 스키마의
  **변동 가능성이 높다.**
- 유형은 "카드에 어떤 요약을 담을지 결정하는 키"이므로 **카드당 정확히 1개**만
  배정된다. 다중 단서 요구는 유형이 아니라 키워드(다중)로 해결한다.

## 결정 (Decision)

- 요약 필드는 `CardSummary` 인터페이스 + 유형별 불변 `record` 9종의
  **다형성 구조**로 설계한다.
- DB에는 단일 **JSON 컬럼**으로 저장하고, Jackson `@JsonTypeInfo` /
  `@JsonSubTypes`로 타입 정보를 JSON에 포함시켜 역직렬화 시 올바른 record로
  복원한다.
- 변환 책임은 JPA `AttributeConverter`(`CardSummaryConverter`)가 전담한다
  (SRP — 엔티티는 데이터 보유, 컨버터는 변환).
- 검색용 키워드는 요약과 분리해 `@ElementCollection`으로 다중 저장한다.

## 고려한 대안 (Considered Options)

1. **유형별 테이블 분리 (JOINED 상속)** — 정규화와 타입 안정성은 최고이나,
   유형 9개 = 테이블 9개. 유형 명칭/필드 변경 때마다 스키마 마이그레이션이
   발생해 "조정 가능" 요구사항과 상충.
2. **EAV (key-value 행 저장)** — 극단적 유연성. 조회 시 조인 폭증과 복잡도
   급증으로 MVP 규모에 과함.
3. **공통 엔티티 + JSON + 다형성 컨버터 (채택)** — DB는 유연, 코드는 타입
   안전. 유형 추가 시 record와 매핑 한 줄만 추가 (OCP). 추후 속성 필터가
   필요해지면 MySQL 8 JSON 가상 생성 컬럼으로 특정 필드만 인덱싱 가능.

## 결과 (Consequences)

### 긍정
- 유형 스키마 변경이 DB 마이그레이션 없이 코드 레벨에서 흡수됨.
- 리스코프 치환 원칙(LSP) — 어떤 구현체든 `CardSummary`로 안전하게 취급.
- 유형이 요약 템플릿을 결정하는 기획 규칙("반드시 1개 배정")과 구조가 일치.

### 부정 / 트레이드오프
- JSON 내부 값 기준 필터링(예: "마감일 있음")은 일반 컬럼 대비 성능 불리 —
  후순위 기능(속성 필터) 도입 시 가상 컬럼 인덱스로 보완 필요.
- 필드 타입 검증이 DB 제약이 아닌 애플리케이션 레이어 책임으로 이동.
- 카드 유형 변경 시 type과 summary를 반드시 함께 교체해야 정합성 유지
  (`InfoCard.changeType`이 이를 강제).
