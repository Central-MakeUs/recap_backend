# ADR-0001: deviceId 기반 익명 식별 및 OAuth 병합 전략

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-07 (소급 기록 — 실제 결정은 2026-06 하순) |
| 관련 | User 엔티티, 추후 OAuth LLD |

## 맥락 (Context)

RECAP MVP에는 타인과 상호작용하는 기능(공유, 댓글 등)이 없고, 데이터가 기기 내
스크린샷 기반의 개인 데이터다. 로그인은 진입 장벽이 되어 "설치 직후 바로
정리 경험"이라는 핵심 가치를 해친다.

다만 로드맵상 다음 기능이 나오는 시점에 영속적 사용자 식별(계정)이 필수가 된다.

- 클라우드 백업 / 기기 교체 복원 (Premium)
- 구독 결제 (결제 복원은 deviceId로 불가능 — 기기 변경 시 결제 이력 유실)
- 팀 공유 (Pro)

또한 App Store 정책상 제3자 소셜 로그인을 하나라도 제공하면 Sign in with
Apple을 함께 제공해야 한다(심사 가이드라인 4.8). 따라서 소셜 로그인 도입 시
카카오 + Apple은 세트로 계획한다.

## 결정 (Decision)

- MVP는 로그인 없이 **클라이언트 생성 UUID를 deviceId로 사용**해 사용자를
  식별한다. 저장 위치는 iOS Keychain / Android EncryptedSharedPreferences.
- `User` 엔티티에 `email`, `oauthProvider`, `oauthId`를 **nullable로 미리
  마련**하고, 로그인 도입 시 **기존 deviceId 레코드에 OAuth 정보를 병합
  (merge)** 한다. 익명 시절 정리한 카드 데이터는 그대로 계정에 승계된다.
- 재병합 시도(이미 OAuth가 연결된 유저)는 예외로 차단해 정합성을 보호한다
  (`User.linkOauth`).

## 고려한 대안 (Considered Options)

1. **클라이언트 생성 UUID (채택)** — 플랫폼 무관 동일 식별 체계, 서버 관점
   단순. 앱 삭제 시 유실은 감수.
2. **플랫폼 제공 ID (IDFV / ANDROID_ID)** — 구현은 빠르나 앱 삭제·공장 초기화
   시 값 변경, 플랫폼 정책 변경에 취약, 추후 계정 연동 시 충돌 처리 복잡.
3. **처음부터 로그인 강제** — 식별은 확실하나 MVP 진입 장벽. 상호작용 기능이
   없는 현 단계에서 얻는 이점이 비용을 정당화하지 못함.

## 결과 (Consequences)

### 긍정
- 설치 직후 무가입 사용 가능 (활성화 경험 극대화).
- OAuth 필드가 준비되어 있어 로그인 도입 시 스키마 변경 없이 값만 채움 —
  기존 코드 변경을 최소화하는 확장(OCP 지향).
- 익명 → 계정 전환 시 데이터 승계가 자연스러움.

### 부정 / 트레이드오프
- 앱 삭제·기기 변경 시 데이터 유실 (결제 도입 전까지는 허용 리스크로 판단).
- deviceId 유일성 보장이 클라이언트 구현에 의존 (FE와 생성/저장 방식 합의 필요).

## 후속 / 미결정
- [ ] 카카오/Apple 로그인 구현 (JWT Access+Refresh, Refresh는 DB 저장,
      검증 방식은 "앱이 토큰 수신 → 서버 검증" 방식으로 확정됨)
- [ ] 병합 시나리오 상세: 로그인한 계정이 이미 다른 deviceId와 연결된 경우의
      처리 (기기 A 익명 사용 → 기기 B에서 로그인 → 기기 A에서 같은 계정 로그인)
