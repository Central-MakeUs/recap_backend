## 맥락 (Context)

Apple은 Android 네이티브 SDK를 제공하지 않아, Android의 Apple 로그인은
웹 기반 OAuth 플로우를 거친다. 이 경우 Apple은 인증 결과를 앱이 아니라
서버(Return URL)로 직접 전달하고, 서버는 처리 후 딥링크(커스텀 URI 스킴)로
앱에 제어권을 넘겨야 한다.

RFC 8252(OAuth 2.0 for Native Apps)는 커스텀 URI 스킴의 근본적 한계를
명시한다: 여러 앱이 동일한 스킴을 등록할 수 있어, 어떤 앱이 실제로 응답을
받을지 운영체제가 보장하지 않는다. 즉 악성 앱이 우리와 동일한 스킴
(`com.cmc.recap://`)을 등록해두면 콜백을 가로챌 수 있다.

## 결정 (Decision)

서버가 발급한 딥링크에 **실제 accessToken/refreshToken을 직접 담지 않는다.**
대신 1회용·60초 유효 `exchangeCode`만 담아 전달하고, 앱은 이 코드를 별도
백채널 POST(`/api/v1/auth/apple/android/exchange`)로 제출해야만 실제 토큰을
받을 수 있다. `exchangeCode`는 서버 인메모리 캐시(Caffeine)에 토큰과
매핑되어 보관되며, 사용 즉시 폐기된다.

## 고려한 대안 (Considered Options)

1. **딥링크에 실제 토큰을 직접 담기 (기각)** — 구현이 가장 단순하지만,
   커스텀 스킴이 가로채이는 순간 살아있는(14일 유효) refresh token까지
   그대로 탈취당한다. 또한 토큰이 URL에 실리면 OS 로그, 브라우저 히스토리
   등에 노출될 위험도 있다.
2. **exchangeCode + 백채널 교환 (채택)** — 가로채이더라도 공격자는 60초
   이내에 교환 요청까지 먼저 성공시켜야 하는 좁은 시간창만 갖는다. 구현
   비용은 엔드포인트 하나와 짧은 TTL 캐시 정도로 크지 않다.
3. **완전한 PKCE 도입 (보류)** — 인가 코드 가로채기 공격의 표준 대응책이나,
   Android 클라이언트에서 code_verifier 생성·보관 등 추가 암호화 구현이
   필요해 지금 범위에서는 과함. exchangeCode 방식으로 실질적 위험은 충분히
   낮아진다고 판단해 후속 과제로 남긴다.

## 결과 (Consequences)

### 긍정
- 딥링크 가로채기 공격의 피해 범위가 "살아있는 토큰 즉시 탈취"에서
  "60초 내 경쟁적 교환 요청"으로 대폭 축소됨.
- 구현 비용이 낮음 — 별도 암호화 로직 없이 캐시 기반으로 해결.

### 부정 / 트레이드오프
- 왕복이 한 단계 늘어남(딥링크 수신 → 즉시 백채널 POST) — Android
  클라이언트가 지연 없이 교환 요청을 호출해야 함.
- exchangeCode 캐시가 인메모리라 서버가 여러 인스턴스로 수평 확장되면
  캐시 공유 문제가 생김(현재 단일 EC2 인스턴스 전제에서는 문제없음).

## 후속 / 미결정
- [ ] 서버 인스턴스가 2대 이상으로 늘어나는 시점에 exchangeCode 저장소를
      Redis 등 공유 저장소로 전환 검토
- [ ] PKCE 도입 여부는 Android 클라이언트 개발 여력에 따라 재검토
