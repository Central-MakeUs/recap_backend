# ADR-0003: 원본 스크린샷 이미지 미저장 (텍스트 요약 완결)

> Architecture Decision Record. 하나의 중요한 의사결정과 그 이유를 기록한다.

| 항목 | 값 |
| --- | --- |
| 상태 | Accepted |
| 날짜 | 2026-07-07 (소급 기록 — 실제 결정은 2026-07 초) |
| 관련 | InfoCard, ADR-0002 |

## 맥락 (Context)

- 분석 파이프라인은 **클라이언트가 OCR을 수행하고 텍스트(JSON)만 서버로
  전송**하는 구조로 확정됨. 서버의 AI(LLM) 분석은 텍스트 기반이므로 분석
  목적으로는 이미지가 필요 없다.
- 당초 "이미지 자체가 정보인 카드"(디자인/레퍼런스)는 원본 보관이 필요하다는
  논의가 있었으나, **디자인/레퍼런스 유형이 기획에서 폐기되고 '기타'로 통합**
  되면서 이미지형 카드 개념 자체가 사라졌다.
- 원본 삭제와 카드 생명주기는 **완전 독립**으로 결정됨 — 사용자가 갤러리에서
  원본을 지워도 RECAP 카드는 유지된다. "두 번 삭제해야 하는" UX 역효과와
  "갤러리 정리 시 카드까지 증발하는" 신뢰 하락을 모두 회피하기 위함.

## 결정 (Decision)

서버는 원본 스크린샷 이미지를 **받지도, 저장하지도 않는다.**
카드는 AI가 생성한 텍스트 요약(CardSummary)과 키워드만으로 완결된다.
`InfoCard`에는 이미지 관련 필드(`originalImageKey`, 업로드 상태 등)를 두지
않는다.

## 고려한 대안 (Considered Options)

1. **전체 카드 원본 저장 (S3 + presigned URL)** — 원본 재확인 요구에
   대응 가능하나, 프라이버시 우려 재점화, 스토리지 비용, t3.small 인스턴스의
   이미지 트래픽 부담. 확정되지 않은 요구를 위한 선제 구현(YAGNI 위반).
2. **하이브리드 (이미지형 카드만 저장)** — 유형 폐기로 전제가 소멸.
3. **완전 미저장 (채택)** — 가역성이 비대칭적: 나중에 저장 기능을 "추가"하는
   비용 < 만들어둔 저장 인프라와 데이터를 "제거"하는 비용. 되돌리기 쉬운
   방향을 우선.

## 결과 (Consequences)

### 긍정
- 프라이버시 리스크 원천 차단 (서버에 사용자 이미지가 존재하지 않음).
- 스토리지 비용 0, 서버 대역폭 부담 없음.
- 갤러리 원본 삭제·기기 변경과 무관하게 텍스트 카드 유지.

### 부정 / 트레이드오프
- "카드 상세에서 원본 다시 보기" 요구가 실사용에서 확인되면 기능 추가 필요
  (예: AI 요약 오류를 원본으로 검증하고 싶은 경우 — 추정 시나리오).
- 대응책: API 응답 DTO는 추후 `imageUrl` 같은 선택적 필드를 얹어도 기존
  클라이언트가 깨지지 않도록 하위 호환 확장 여지를 유지한다.
