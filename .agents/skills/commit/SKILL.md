# commit

RECAP 서버의 변경사항을 **`type: 한글 제목(#이슈번호)`** 형식으로 커밋한다.
타입만 영문, 제목과 본문은 한글로 작성한다.

> 이 형식은 RECAP 프로젝트의 확정 컨벤션이다. 다른 형식을 임의로
> 섞어 쓰지 않는다.

## 핵심 원칙

- 사용자가 커밋을 요청했을 때만 커밋한다. 기능 구현 도중에는 사용자 확인
  없이 커밋하지 않는다.
- **예외**: `docs/` 하위 문서(ADR, LLD, policy, 템플릿) 변경은 `CLAUDE.md`의
  커밋 정책에 따라 확인 없이 자율 커밋할 수 있다. 그 외(소스 코드, 설정,
  CI, `CLAUDE.md` 자체)는 항상 사용자 확인 후 커밋한다.
- 어떤 변경을 어떤 이슈 단위로 묶을지 모호하면 추측하지 말고 사용자에게
  물어본다.
- 관련 없는 변경을 한 커밋에 섞지 않는다. 논리적 단위로 나눈다.
- `.agents/`는 로컬 작업물로 취급한다. 사용자가 명시하지 않으면
  스테이징하지 않는다. (`docs/policy/`는 위 자율 커밋 대상에 포함되므로
  이 규칙에서 제외한다.)

## 절차

1. `git status --short --branch`로 현재 브랜치와 변경 파일을 확인한다.
2. `git diff`와 필요 시 `git diff --staged`로 변경 내용을 읽는다.
3. 변경을 논리적 단위로 나눈다. 관련 없는 변경은 제외한다.
4. 아직 검증하지 않았다면 관련 테스트/빌드 실행 여부를 위험도에 맞춰
   판단한다. (PR 전에는 `./gradlew build` 통과가 필수다.)
5. 스테이징은 의도한 파일만 한다.
6. 아래 형식으로 커밋한다.

## 커밋 메시지 형식

```
<type>: <한글 제목, 마침표 없음>(#<issue-number>)

<한글 본문 — "왜" 변경했는지. 무엇을 했는지보다 이유 중심>
```

이슈 번호가 없는 순수 잡일(docs 자율 커밋 등)은 괄호를 생략한다.

```
<type>: <한글 제목>
```

### type (영문 고정)

- `feat` 새 기능
- `fix` 버그 수정
- `docs` 문서 (ADR/LLD/README 등)
- `refactor` 동작 변경 없는 구조 개선
- `test` 테스트 추가/수정
- `chore` 빌드/설정/그 외 잡일

## 예시

```
feat: 카카오 OAuth 로그인 엔드포인트 추가(#12)

LLD-0001 기준으로 provider 토큰 검증과 JWT 발급까지 처리한다.
단일 엔드포인트(/api/auth/oauth/{provider}/login) 구조를 채택했다.
```

```
docs: LLD-0001 카카오/Apple 소셜 로그인 확정
```

## 하지 말 것

- 어떤 이슈에 연결할지 임의로 정하기 → 모르면 묻는다.
- 미완성/실패하는 테스트를 "통과한다"고 본문에 쓰기.
- 관련 없는 변경 묶기.
- 사용자 변경이나 생성물을 임의로 되돌리기.
- ADR-0005(JPA 컴포넌트 DI 배제)처럼 이미 확정된 코드 규칙과 어긋나는
  코드를 별도 언급 없이 커밋하기.
