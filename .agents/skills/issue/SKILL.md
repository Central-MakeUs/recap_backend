# issue

RECAP 서버 작업을 시작하기 전에 GitHub Issue를 생성하고, 필요하면
`{type}#{issue-number}` 브랜치를 만든다.

## 핵심 원칙

- 이슈 제목·본문은 한글로 작성하되, **제목 형식은 반드시 기존
  `.github/ISSUE_TEMPLATE`를 따른다** (예: `🌱[FEAT] 회원 가입 플로우`,
  `⛑️[BUG] 로그인 실패`). 템플릿에 없는 새 제목 규칙을 임의로 만들지 않는다.
- 작업 범위가 크면 한 이슈에 몰아넣지 말고 사용자에게 분리안을 제안한다.
- 정책 미결정 사항은 결정된 것처럼 쓰지 말고 `확인 필요`로 남긴다.
- 아키텍처 관련 이슈라면 관련 ADR(`docs/adr/`)이 있는지 먼저 확인하고,
  없다면 "ADR 작성이 먼저 필요할 수 있음"을 이슈 본문에 언급한다.
- 이미 열린 관련 이슈가 있으면 새로 만들기 전에 재사용 여부를 확인한다.
- 브랜치는 사용자가 요청했거나 작업을 바로 진행할 때만 만든다.

## 절차

1. `gh issue list --state open --limit 30 --json number,title,labels`로
   중복 이슈를 확인한다.
2. 어떤 이슈 템플릿(기능 요청/버그 리포트)에 해당하는지 판단하고, 해당
   템플릿의 제목 접두어와 본문 섹션을 그대로 사용한다.
3. `gh issue create --title "<template 제목>" --body-file <tmpfile>`로
   이슈를 만든다. (`--template` 플래그로 기존 템플릿 파일을 지정할 수 있으면
   우선 사용한다.)
4. 라벨이 필요하면 기존 라벨을 먼저 확인한다: `gh label list --limit 100`.
5. 사용자가 브랜치 생성을 요청했거나 바로 작업을 이어가면
   `git switch -c {type}#{issue-number}`를 실행한다 (예: `feat#12`).
6. 생성한 이슈 번호, URL, 브랜치명을 사용자에게 보고한다.

## 라벨 가이드

RECAP 저장소의 실제 라벨 목록(`gh label list --limit 100`, 2026-07-08 확인
기준)은 아래와 같다. 목록이 바뀌었을 수 있으니 사용 전 다시 확인한다.

- 기능 구현: `Feature` / 기능 수정(구현): `Fix` / 버그(정상 동작 안 함): `Bug`
- 문서: `Docs` / 리팩터링: `Refactor` / API 통신: `API`
- 빌드·배포·CI/CD·설정 파일 작업: `Chore` / 개발 세팅: `Setting`

`논의`(question)에 대응하는 라벨은 없다 — 임의로 만들지 않는다.

`젤로`/`조이`/`예니`/`올리버`/`민물`은 담당자 태그용 라벨로 보인다(작업
분류 라벨이 아님). 실제 용도가 불확실하면 사용자에게 확인한다.

## 하지 말 것

- 중복 이슈를 확인하지 않고 새 이슈 만들기.
- 기존 `.github/ISSUE_TEMPLATE` 형식을 무시하고 임의 제목/본문 만들기.
- 한 이슈에 명세, 구현, 배포, 정책 결정을 모두 섞기.
- 사용자가 원하지 않았는데 main에서 바로 작업하기.
- 이슈 생성만 요청했는데 커밋이나 PR까지 진행하기.
